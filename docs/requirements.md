# 要件定義書（study-pilot） v0.1

## 目的
- 過去問のある試験を対象に、学習範囲をタスクへ分解し、完了条件付きで提示・管理・可視化できるシステムを提供する。
- 学習者が「何を・どの順で・どこまでやれば合格に近づくか」を明確にし、効率的かつ計画的に学習を進められるようにする。

## 背景
- 独学者は「学習範囲の把握」「タスク分解」「進捗管理」に課題を抱えやすい。
- 既存ツールは点在しており、「範囲→タスク→進捗→弱点再学習」の一貫管理が不十分。
- 本書は concept.md v0.3 を前提に、MVPとして実装すべきシステム要件を定義する。

## スコープ

### 対象
- 個人学習者（単一ユーザー）を想定。
- 過去問のある試験を対象（例：国家資格、公開学力試験）。

### 対象範囲
- 試験情報の収集・確定・参照（LLMヒアリング前提）。
- 試験範囲のタスク分解（完了条件の自動付与・編集を含む）。
- タスクの優先度付け。
- 学習実行の記録（システム内＝自動判定／システム外＝手動記録）。
- 進捗可視化と弱点抽出。
- 検索・フィルタ、最小限の変更履歴。
- 自動保存と復元（機能として提供）。

### 対象外
- 学習者コミュニティや教材作成・共有機能。
- 過去問本文の解析や長文解説の自動生成。
- 音声入力や手書き採点などの高度機能。
- 複数ユーザー／チーム利用。
- クラウド同期。

## 機能要件

### 試験情報のインタラクティブ収集・確定・参照
- 機能
  - LLM と対話しながら受験意図・試験情報をヒアリングし、必要項目を漏れなく収集する。
  - 不足・曖昧な回答を検知し、追加質問で補完する。
  - 収集結果をユーザーが確認し「確定」状態にできる。
  - 最小限の参照ビューで最新の確定情報を確認できる。
- 入力
  - ユーザーの自然言語回答（試験名、合格基準、範囲の大項目、過去問の有無 など）。
- 出力
  - 収集内容の要約（未確定項目の明示を含む）。
  - 確定済みの試験情報（参照ビュー）。
- 振る舞い（状態）
  - draft（下書き）→ 確認 → confirmed（確定）。
  - 未充足項目がある場合は確定をブロックし、追質問を促す。
  - 中断時は draft を保持し、続きから再開できる。
- 受け入れ基準
  - 必須項目が未充足のまま confirmed に遷移できない。
  - confirmed 時に「収集要約」「未確定項目の有無」が出力される。
  - 参照ビューで必須情報を一画面で確認できる。

### 試験範囲のタスク分解
- 機能
  - 試験範囲を「トピック→タスク」に自動分解する。
  - 各タスクに初期の完了条件（テンプレ＋しきい値）を同時付与する。
  - 自動生成結果（タスク名＋完了条件）をユーザーが手動で調整できる（追加／削除／名称変更／分割・統合／条件編集）。
- 入力
  - 分解対象の範囲、分解レベル（深さ）などの指示。
- 出力
  - 完了条件付きのタスクツリー（階層一覧）。
- 振る舞い（状態）
  - draft → 確認 → confirmed（確定時に「条件未設定タスク＝0」を保証）。
  - 確定後も再編集可能（変更履歴は保持）。
- 受け入れ基準
  - 生成直後にすべてのタスクが完了条件を持つ（空なし）。
  - 曖昧・不十分な条件（例：「理解した」）は検出・警告し、確定をブロック。
  - 粒度のヒント（1セッションで完了可能）を提示する。

### 優先度付け
- 機能
  - 各タスクに優先度（例：A/B/C）を付与し、並び替え・絞り込みに利用できる。
  - 優先度の根拠（頻出度・配点・コア領域など）をメモとして添えられる。
- 入力
  - タスク選択、優先度、任意の根拠メモ。
- 出力
  - 優先度でソート／フィルタされたタスクリスト。
- 振る舞い（状態）
  - 優先度未設定タスクは「要設定」ビューで一括確認できる。
- 受け入れ基準
  - 優先度未設定タスクを検出・一覧化できる。
  - 優先度更新は即時表示に反映される。

### 学習実行の記録（自動／手動）
- 機能
  - システム内タスクは実行結果に基づき、完了／未完了・正答状況を自動記録する。
  - システム外タスク（手書き図・実機操作など）はユーザーが手動で完了記録できる（任意メモ可）。
- 入力
  - 自動：タスク実行結果（正誤・スコア等）。
  - 手動：タスク選択、完了チェック、任意メモ。
- 出力
  - タスク状態（未着手／進行中／完了）、直近の実行履歴。
- 振る舞い（状態）
  - すべての記録にタイムスタンプを付与。
  - 途中離脱は「進行中」で保存し、再開可能。
- 受け入れ基準
  - 自動・手動いずれの記録も即時に状態へ反映される。
  - 手動記録は最小入力（完了チェックのみ）で保存可能。

### 進捗可視化と弱点抽出
- 機能
  - 進捗指標（完了タスク率、優先度帯別達成度、週あたり完了数）を表示する。
  - 未達・低正答のタスクを弱点として抽出し、再学習対象として提示する。
- 入力
  - タスク状態、実行履歴。
- 出力
  - ダッシュボード（期間フィルタ対応）、推奨タスクリスト（例：優先度高×未達、正答率低下中）。
- 振る舞い（状態）
  - 期間（週／月）切替で再計算される。
  - 指標の算出ロジックには簡潔な注釈を添える。
- 受け入れ基準
  - 最新の記録がダッシュボードに即時反映される。
  - 推奨タスクから直接着手できる動線がある（一覧→詳細→実行）。

### タスク検索・フィルタ
- 機能
  - タイトル、範囲、トピック、優先度、状態などで検索・絞り込みできる。
- 入力
  - 条件指定（基本AND）。
- 出力
  - 該当タスクリスト。
- 受け入れ基準
  - 最低限の複合フィルタが可能。

### 変更履歴（最小）
- 機能
  - 主要フィールド（優先度、完了条件、状態）の変更履歴を保持し、直近の変更を参照できる。
- 入力
  - 変更内容。
- 出力
  - 直近N件の履歴ビュー（NはMVP内で定義）。
- 受け入れ基準
  - 現在値と直近履歴が参照可能。

### 自動保存・復元（機能要件）
- 機能
  - 進捗・タスク編集を自動保存する。
  - アプリ異常終了後に前回の状態を復元できる。
- 入力
  - 保存トリガ（編集確定など）、復元操作。
- 出力
  - 保存結果・復元後の現在状態。
- 受け入れ基準
  - 保存失敗時はユーザーに通知し、再試行手段を提示する。

## ユーザーシナリオ

### S1: 試験登録と範囲確認
- 目的
  - 学習対象（試験）と合格条件を明確化し、以降の分解の前提を固める。
- 前提
  - 過去問のある試験であること。
- トリガー
  - ユーザーが試験名を入力し、範囲・合格条件を登録する。
- 手順（ハッピーパス）
  - 試験名を入力。
  - 範囲の大項目と合格条件（合格点・配点など）を回答。
  - 登録内容を確認ビューで参照。
- 代替/例外
  - 必須項目欠落 → 登録不可メッセージと不足項目の提示。
- 成功条件
  - 登録済み試験情報が一貫形式で参照可能。

### S2: タスク分解（完了条件付き、自動生成＋手動調整）
- 目的
  - 範囲を「トピック→タスク」へ分解し、実行可能な単位にする。
- 前提
  - S1 が完了。
- トリガー
  - 「タスク分解を実行」を選択。
- 手順（ハッピーパス）
  - 分解レベル等のパラメータを指定し、自動分解。
  - 出力されたタスクツリーを確認し、名称・粒度・完了条件を必要に応じて編集。
- 代替/例外
  - 範囲外タスク → 削除。
  - 粒度過大／過小 → 分割／統合。
- 成功条件
  - すべてのタスクが完了条件を持ち、1セッションで完了可能な粒度。

### S3: 優先度付け
- 目的
  - 合格に直結するタスクから着手できるよう順序付けする。
- 前提
  - S2 が完了。
- トリガー
  - 「優先度付け」を開始。
- 手順（ハッピーパス）
  - 各タスクに優先度（A/B/C）を設定し、根拠メモを入力。
- 代替/例外
  - 未設定タスク → 「要設定」ビューで一括処理。
- 成功条件
  - 優先度でソート／フィルタが可能、未設定タスクが残存しない。

### S4: 学習の実行と記録（システム内＝自動判定）
- 目的
  - 自動判定可能タスクの結果から進捗を自動記録する。
- 前提
  - S3 が完了。
- トリガー
  - ユーザーがシステム内タスクを開始。
- 手順（ハッピーパス）
  - タスクを実行し、結果（スコア/正誤）を取得。
  - しきい値達成で「完了」に自動更新、未達なら継続/再試行へ。
- 代替/例外
  - 途中離脱 → 「進行中」で保存。
- 成功条件
  - 実行直後に状態・履歴が自動更新（タイムスタンプ付き）。

### S5: 学習の実行と記録（システム外＝手動記録）
- 目的
  - 手書き図や実機操作など、システム外タスクの達成を軽負担で記録する。
- 前提
  - 対象タスクが手動記録対象。
- トリガー
  - 手動記録フォームを開く。
- 手順（ハッピーパス）
  - 完了チェックと任意メモ（成果物場所など）を入力して保存。
- 代替/例外
  - 入力負荷が高い → 必須は最小（完了チェックのみ）。
- 成功条件
  - 1分以内で記録完了し、ダッシュボードに反映。

### S6: 進捗確認と次タスクの選択
- 目的
  - 現在地の把握と、次にやるべきタスクの意思決定を支援する。
- 前提
  - S4/S5 が進行中。
- トリガー
  - ダッシュボードを開く。
- 手順（ハッピーパス）
  - 完了タスク率、優先度別達成度、週あたり完了数、弱点（低正答/未達）を確認。
  - 推奨タスク（優先度A未達や弱点補強）から次の1件を選び着手。
- 代替/例外
  - 推奨に従わず任意選択 → 記録は同様に反映。
- 成功条件
  - 指標が最新データで計算され、弱点抽出と推奨が表示。

### S7: 再学習（弱点補強）
- 目的
  - 未達タスクの再試行により定着を図る。
- 前提
  - S6 の弱点抽出がある。
- トリガー
  - 再学習用「再テスト」開始。
- 手順（ハッピーパス）
  - インターバル後に再試行し、しきい値達成を目指す。
- 成功条件
  - 過去履歴と区別できる形で完了が記録（再試行回数、直近達成など）。

### シナリオ間の連携
- S1 → S2 → S3：範囲確定から分解・優先度設定。
- S3 → S4/S5：実行と記録（自動/手動）。
- S4/S5 → S6 → S7：進捗確認 → 弱点補強 → 再学習のループ。

### 受け入れ基準（横断）
- 優先度Aのタスクで「登録→分解（条件付）→実行→記録→可視化」が一貫して成立する。
- ダッシュボードで少なくとも以下を確認可能
  - 完了タスク率
  - 優先度A達成度
  - 週あたり完了数
  - 弱点一覧（未達/低正答）

## 非機能要件（MVP）

### 性能・応答性
- ユーザー操作（参照、一覧、記録、フィルタ）は 1 秒以内に応答する。
- タスク分解など LLM 処理は待機が発生してもよいが、処理中であることを明示する（進行状況表示／キャンセル・再試行可）。

### 運用・保守性
- エラーログはデバッグ用に保存し、ユーザーへの影響を最小化する。

### セキュリティ・プライバシー
- データはすべてローカルに保存する。
